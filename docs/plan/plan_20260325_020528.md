# 🔍 Agent Loop & Bottleneck Analysis Report

## Executive Summary

The `obsidian-forge` codebase has several significant performance bottlenecks in its agent loop, primarily due to **sequential processing** where parallel/concurrent execution would be beneficial. The main bottlenecks are:

1. **Sequential AI calls** in note processing (4+ blocking AI calls per note)
2. **Sequential file I/O** in graph strengthening and MOC updates
3. **Synchronous PDF conversion** in the watcher
4. **No batching** for file system operations

---

## 📍 Bottleneck Analysis by Component

### 1. 🔴 Critical: Sequential AI Calls in `process_one()` 

**File:** `src/notes.rs` (lines 79-135)

**Problem:** Each note triggers **4 sequential AI API calls**:

```rust
// Line 94: AI Call #1 - Summarization
let summary = ollama.summarize(&body, 200).await.unwrap_or_default();

// Line 99: AI Call #2 - Question generation
let questions: Vec<String> = ollama.generate_json(&q_prompt).await.unwrap_or_default();

// Line 106-107: AI Call #3 - Tag generation  
let gen_tags: Vec<String> = ollama.generate_json(&t_prompt).await...

// Line 114: AI Call #4 - Classification (potentially another AI call)
let (category, subcategory, detail) = classify_by_title_or_ai(&title, &body, &ollama, &prompts).await;
```

**Impact:** 
- If each AI call takes ~2-5 seconds, a single note takes **8-20 seconds** to process
- Processing 10 notes sequentially = **80-200 seconds**

**Solution Direction:**
```rust
// Parallelize independent AI calls using tokio::join! or futures::join!
let (summary, questions, gen_tags) = tokio::join!(
    ollama.summarize(&body, 200),
    ollama.generate_json::<Vec<String>>(&q_prompt),
    ollama.generate_json::<Vec<String>>(&t_prompt),
);
```

### 2. 🔴 Critical: Sequential File Processing in `process_all()`

**File:** `src/notes.rs` (lines 33-77)

**Problem:** Files are processed one-by-one:

```rust
// Lines 53-63: Sequential markdown processing
for entry in fs::read_dir(&inbox)? {
    let path = entry.path();
    if is_markdown(&path) {
        process_one(&path, config, vault_root).await?;  // ← Blocks on each file
    }
}
```

**Impact:** 10 notes × 15 seconds each = **150 seconds total**

**Solution Direction:**
```rust
// Use futures::stream for concurrent processing
use futures::{stream, StreamExt};

let files: Vec<_> = fs::read_dir(&inbox)?
    .filter_map(|e| e.ok())
    .filter(|e| is_markdown(&e.path()))
    .collect();

stream::iter(files)
    .map(|entry| process_one(&entry.path(), config, vault_root))
    .buffer_unordered(5)  // Process up to 5 concurrently
    .collect::<Vec<_>>()
    .await;
```

## 🛠️ Implementation Plan

### Phase 1: Parallelize AI Calls (Highest Impact)

**File:** `src/notes.rs`

1. Refactor `process_one()` to use `tokio::join!` for independent AI calls
2. Add `futures` crate dependency if not present
3. Estimated improvement: **4× speedup** for AI operations

### Phase 2: Concurrent Note Processing

**File:** `src/notes.rs`

1. Add `futures` and `tokio-stream` dependencies
2. Refactor `process_all()` to use `buffer_unordered()`
3. Add configurable concurrency limit (e.g., `config.ai.max_concurrent`)
4. Estimated improvement: **N× speedup** (N = concurrent limit)

### Phase 3: Async PDF Conversion

**File:** `src/converter.rs`

1. Replace `std::process::Command` with `tokio::process::Command`
2. Update `watcher.rs` to handle async conversion
3. Estimated improvement: **Non-blocking watcher**

### Phase 4: Parallel Graph Operations

**File:** `src/graph.rs`

1. Add `rayon` dependency
2. Parallelize `scan_all_projects()` and `scan_project()`
3. Cache `fs::read_dir(&zk_path)` results to avoid repeated reads
4. Estimated improvement: **2-4× speedup** for large vaults

### Phase 5: Parallel MOC Updates

**File:** `src/moc.rs`

1. Use `rayon` for `update_all_mocs()`
2. Estimated improvement: **2× speedup** for many projects
